## Why blocked (two independent showstoppers)

### Issue 1 — host-shell dev server stuck on a stale compile error

- Webpack-dev-server overlay reports, on every reload, the same TS2307 error pointing at `apps/host-shell/src/app/app.routes.ts:35:14`:
  ```
  Cannot find module './core/debug/http-error-tester.component'
  ```
- The on-disk file `C:\Falcon\Falcon\falcon-web-platform-ui\apps\host-shell\src\app\app.routes.ts` does NOT contain that reference. Line 35 is the `falcon-ui-showcase` route. The `core/debug/` folder does not exist.
- The dev server (PID 63240) is therefore serving a partial / dead JS bundle: the login form renders (probably from cache) but the submit handler is non-functional. Clicking "Login" spins forever with no auth network call.
- Effect: it is impossible to drive the login UI to completion through host-shell:4200 without a dev-server restart, which is outside QA scope.

### Issue 2 — frontend env wired to the cloud, not the local docker stack

- `apps/host-shell/src/environments/environment.ts` has:
  ```ts
  baseURLIdentityGateway: 'https://auth.falconhub.space/api/'
  baseURLCoreGateway:     'https://core-api.falconhub.space/'
  baseURLSystemGateway:   'https://system-api.falconhub.space/'
  baseURLPes:             'https://pes-api.falconhub.space/'
  ```
- Comment in the file (lines 18-28) explicitly says: "Local-docker stack (`docker compose up -d` at Falcon/Falcon/docker-compose.yml). Swapped from staging gateways on 2026-05-16 …" — but the URLs in the actual exports point at the **non-staging cloud** hosts (`auth.falconhub.space`, not the empty `baseURL: ''`). So the FE will not talk to the local docker Identity at port 7777 regardless of dev-server state.
- I confirmed by API probe: `POST https://auth.falconhub.space/api/auth/login` with `accowner / Admin@1234` returns `401 {"errorMessages":["Invalid username or password."]}`. The cloud auth does not know the docker-seeded test users.
- Effect: even if Issue 1 were fixed, the seeded `accowner / accadmin / accuser` accounts cannot be used to drive the FE because the FE is calling out to a different identity provider altogether.

## What I did manage to verify

- **Backend identity service reachable on docker:** Identity at `0.0.0.0:7777` is listening (per `docker ps`). However, the FE never reaches it — see Issue 2.
- **Front-end ports listening:** PID 63240 holds `[::1]:4200`, `[::1]:4204`, `[::1]:4301`. They are IPv6-only (curl over IPv4/IPv6 from outside Chrome cannot connect); the Chrome browser reaches them via its own localhost resolution.
- **Login form renders.** The host-shell login page (`http://localhost:4200/#/login`) loads its T2 Falcon hero panel + form. Form input works (username and password fields accept text). Submit does nothing because the JS bundle is dead.
- **Compile-error overlay is recurring.** Webpack reports the same `http-error-tester.component` error on every navigation, indicating it is reading from a stale source map / cache, not the actual file on disk.

## Honest issues encountered

1. **Stack state claim in the brief was optimistic.** The brief said "Frontend: host-shell:4200 + admin-console:4204 + management-console:4301 (all listening, PID 63240)". Listening, yes — but the host-shell webpack-dev-server has been in a failed-compile loop the entire session, serving a broken bundle. "Listening" ≠ "serving working JS".
2. **Environment configuration drift.** `environment.ts` has comments saying the dev env was switched to the local docker stack on 2026-05-16, but the URL constants still point at cloud hosts. Either the swap was reverted and the comments are stale, or the swap was only partially applied (the comment was updated but the URLs weren't). This is the kind of thing that bites every QA run until reconciled.
3. **The `http-error-tester.component` ghost.** The on-disk routes file does not reference this component anywhere, yet webpack keeps reporting it as a TS2307 error. This points at one of: a stale `.nx/cache` entry, a Jest/typecheck process holding an old AST, a different routes file being read from a path I haven't found, or a long-lived hot-update fragment in the dev server. A focused dev-side investigation would untangle this; from QA's seat, it just means "the dev server is poisoned, needs a restart."

# Backend-level supplementary verification (option B) — RAN + PASSED

**Date:** 2026-05-16 (later in the same session, after the FE blockers were documented)
**Method:** Direct API calls to Identity (port 7777) for login + PES (port 5296) for `/pes/authorize`. Bypasses the broken FE entirely.
**Verdict:** **✋ 21/21 PASS — every PES decision matches the dataset's predictions perfectly.**

## What changed in the approach

Instead of driving the broken FE through Chrome, the verification:

1. POST `http://localhost:7777/api/auth/login` with `{ username, password }` for each test user → gets back a `ServiceOperationResult<LoginStepResponse>` with `result.tokens.accessToken` (JWT, RS256, signed by Zitadel at issuer `http://localhost:8080`)
2. Extracts `JWT.sub` (the Zitadel user-id) from the access token's payload
3. POST `http://localhost:5296/pes/authorize` with the structured `AuthRequest` body:
   ```json
   {
     "sub": { "kind": "u:<JWT.sub>@<tenant-id>" },
     "obj": { "kind": "<resource>" },
     "actions": ["<action>"]
   }
   ```
4. PES returns `{ "results": { "<action>": true|false } }` — true=allow, false=deny

## PES response shape discovery

The brief in §"Why blocked" mentions the dataset's `acc.services/view` rule for acc-admin is `deny` per `BuiltInRoleCatalog.cs:227`. The runtime test confirms this:

```bash
# accadmin → acc.services / view
POST http://localhost:5296/pes/authorize
{
  "sub":  { "kind": "u:373185576859336714@test-tenant-001" },
  "obj":  { "kind": "acc.services" },
  "actions": ["view"]
}
# → { "results": { "view": false } }  ← deny, matches dataset ✅
```
