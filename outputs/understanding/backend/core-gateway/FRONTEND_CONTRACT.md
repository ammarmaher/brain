# Core Gateway — Frontend Contract

## What This Is

The **entry point** for all client-facing browser traffic in the Falcon Web Platform UI. The frontend always talks to the Core Gateway (`https://localhost:7038` in dev, `<core-gateway>` in prod) — never directly to backend services.

## URL Convention

```
<core-gateway>/<service-prefix>/<service-internal-path>
```

| Service Prefix | Routes To | Auth |
|---|---|---|
| `/identity/auth/*` | Identity `/api/auth/*` | **Anonymous** |
| `/identity/*` | Identity `/api/*` | ClientOnly + PerTenant |
| `/commerce/*` | Commerce `/api/*` | ClientOnly + PerTenant |
| `/charging/*` | Charging `/api/*` | ClientOnly + PerTenant |
| `/provisioning/*` | Provisioning `/api/*` | ClientOnly + PerTenant |
| `/contactgroup/*` | Contact Group `/api/*` | ClientOnly + PerTenant |

Plus the **3 aggregation endpoints** under `/commerce/accounts/...` and `/commerce/contracts*` which the gateway handles itself (no YARP) — same `ClientOnly` policy.

## Authentication Flow

```
1) POST /identity/auth/login       (Anonymous — no JWT needed)
   body: { username, password }
   → 200 LoginStepResponse { sessionId, stage, requiresOtp, otpExpiresInSeconds, ... }

2) POST /identity/auth/verify-otp  (Anonymous)
   body: { sessionId, code }
   → 200 LoginStepResponse { stage: "Authenticated", tokens: { accessToken, refreshToken, ... } }

3) For all subsequent calls, include:
   Authorization: Bearer <accessToken>

4) POST /identity/auth/refresh-token  (Anonymous)
   body: { refreshToken }
   → 200 AuthenticatedResult { accessToken, refreshToken, ... }
```

Access token lifetime: 1800s (30 min). Refresh before expiry.

## Tenant Resolution

Once authenticated, the client's tenant id is encoded in the JWT. The gateway:
- Reads it via `ICurrentUser.TenantId`
- Injects it into the forwarded request as `X-Tenant-Id` header
- Uses it as the `PerTenant` rate-limit bucket key
- Uses it as the IP allowlist lookup key

Frontend never needs to send tenant id explicitly.

## Status Codes

The client sees the **downstream service's** status code, preserved by YARP / `HttpResponseHandler`. So:
- `GET /commerce/Node/xyz/applications` returning 404 means Commerce returned 404 (`NodeNotFound`)
- `GET /commerce/contracts` (aggregation) returning 502 may mean Commerce or Charging is down (gateway propagates the failing call's status)

## Cross-Service Aggregations (Important)

Use these instead of multiple parallel calls:

| Aggregation Endpoint | Replaces |
|---|---|
| `GET /commerce/accounts/{Id}/hierarchy` | `GET /commerce/accounts/hierarchy` + `GET /identity/user/by-tenant` + `POST /charging/Wallet/get-account-wallets` |
| `GET /commerce/contracts` | `GET /commerce/Contracts` + `GET /charging/Wallet/contract-balance-summaries` |
| `GET /commerce/contracts/{ContractId}` | `GET /commerce/Contracts/{id}` + `GET /charging/Wallet/contract-balance-summaries` + cross-tenant check |

Note the `{Id}` in the hierarchy aggregation is **ignored** — tenant id always comes from JWT. The route param exists for API shape compatibility with the System Gateway version (which uses the tenant id explicitly).

## Headers Frontend Should Pass

| Header | Required? | Notes |
|---|---|---|
| `Authorization: Bearer <jwt>` | yes (except `/identity/auth/*`) | |
| `Accept: application/json` | optional | Framework default |
| `Accept-Language: en` / `ar` | recommended | Drives `ErrorLocalizer` choice |
| `X-Correlation-Id` | optional | Echoed/generated by middleware for tracing |

Frontend should **not** send `X-Tenant-Id` — the gateway sets it.

## Rate Limit

100 requests / 60 seconds per tenant. Exceeded → HTTP 429 with no body. Frontend should backoff and retry.

## IP Allowlist

If the tenant has an allowlist configured and the client's IP doesn't match, all requests get HTTP 403 with `IpNotAllowed`. Frontend should show a friendly message and let the user know to contact their administrator.

## Pre-Signed S3 URLs (Contact Group)

For Contact Group uploads, the workflow involves PUTting **directly to S3**, not through the gateway:
1. `POST /contactgroup/contact-groups/uploads/init` → returns `preSignedUrl` from S3
2. Frontend does `PUT <preSignedUrl>` directly to S3 with the file body
3. `POST /contactgroup/contact-groups/uploads/{uploadId}/complete` to finalize

S3 must have CORS configured to allow the frontend origin.

## CORS

Gateway CORS allows `http://localhost:4200` in dev (per `appsettings.Development.json`). Production must add the production origin.

## OpenAPI

Dev: `https://localhost:7038/openapi/v1.json` (auto-launched in browser). Aggregation endpoints are listed; YARP routes are not in the OpenAPI doc.

## What's NOT Through The Gateway

- Templates service (no `templates-cluster` configured)
- PES / Access service (internal-only)
- Direct S3 PUT for uploads (separate URL flow)

If you need Templates or PES in the future, the gateway route map must be updated.
