*** ADR-005 — Why dual-render path (Shadow DOM + Tailwind/Light DOM variants) ***
*** Status: accepted · Date: 2026-05-16 · Severity-of-reversal: low (drop one variant) ***

# ADR-005 — Why dual-render path (Shadow DOM + Tailwind/Light DOM variants)

## Context

Falcon ships its own cross-framework component library (`@falcon/ui-core`) built on Stencil. Stencil offers two compile targets per component — Shadow DOM (`shadow: true`) or Light DOM (`shadow: false`) — and the two have opposite trade-offs:

| | Shadow DOM | Light DOM |
|---|---|---|
| Style isolation | Perfect — host styles cannot bleed in, component styles cannot bleed out | None — consumer's global CSS wins |
| Tailwind utility classes from consumer | **Blocked** — utilities generated in `apps/admin-console/src/tailwind.css` and `apps/host-shell/src/tailwind.css` do not pierce the shadow boundary | **Free** — utilities cascade in normally |
| Consumer-side class overrides (`class="..."`, `wrapperClass`, `inputClass`) | Limited — only via `::part()` selectors or per-instance token mutation | Native — caller-supplied classes land directly on rendered nodes |
| Token mutation at runtime (Studio) | Works — `--falcon-*` CSS custom properties inherit through Shadow | Works — same custom-property inheritance |

The Falcon library has two real consumer scenarios driving opposite needs:

1. **`apps/admin-console` and `apps/host-shell`** — Tailwind v4 first, every screen consumes utility classes through `class=` and `wrapperClass`. Shadow DOM is a wall here.
2. **Future React, Vue, and third-party-host integrations** — those consumers may not run Tailwind at all and want bulletproof encapsulation so component internals can't be broken by their own global CSS.

A third hard constraint: the Falcon Theme Studio (Wave 2 Phase E) lets a user mutate **one** token at runtime and must see **both** render paths update simultaneously. Drift between paths breaks the Studio. So whatever rendering strategy we pick must keep both paths token-faithful by construction.

The pain to solve is therefore: how do we get Tailwind utility-class consumption AND Shadow encapsulation AND a single token source of truth, per component, without forking the design system into two divergent libraries?

## Decision

**Every Falcon component ships in two variants compiled from a single token contract and exposed through a single Angular wrapper.**

For component `falcon-X`:

| Variant | Stencil tag | Stencil config | Source file | Style source |
|---|---|---|---|---|
| Shadow | `<falcon-X>` | `shadow: true` | `libs/falcon-ui-core/src/components/falcon-X/falcon-X.tsx` | `falcon-X.css` (scoped, reads `var(--falcon-X-*)`) |
| Tailwind / Light | `<falcon-X-tw>` | `shadow: false` | `libs/falcon-ui-core/src/components/falcon-X-tw/falcon-X-tw.tsx` | Tailwind utilities from `libs/falcon-ui-core/src/tailwind/X-tailwind-classes.ts` (also read `var(--falcon-X-*)`) |
| Token contract (SSOT) | — | — | `libs/falcon-ui-core/src/components/falcon-X/falcon-X.tokens.css` (or `libs/falcon-ui-tokens/src/components/X.tokens.css` for early components) | — |

The thin Angular wrapper at `libs/falcon-ui-core/angular-wrapper/components/falcon-X/` renders **one of the two web components** based on a `useTailwind: boolean` input. Default is `true` — the Tailwind/Light variant is the production target, matching admin-console and host-shell's Tailwind v4 setup. Setting `[useTailwind]="false"` switches to Shadow for consumers who need slot-based composition or hard isolation.

Both variants resolve every visual value through the same `--falcon-X-*` token chain that bottoms out in the workspace theme (`libs/falcon/src/theme/falcon-tailwind-tokens.css` → `@theme`). The Stencil-Shadow path is the SSOT for the token contract: the Tailwind variant **mirrors** the same names and **never** invents tokens or hardcodes values. If a Tailwind-side need arises, the contract is extended first, the Shadow CSS adopts the new token, and the Tailwind variant references it — never the reverse. See [feedback_shadow_is_token_ssot](../../../memory/feedback_shadow_is_token_ssot.md).

## Alternatives considered

### A. Shadow-only — rejected

Single variant, `shadow: true` for every component. Best encapsulation, single source per component, no maintenance overhead from a second TSX file.

**Rejected because:**
- Tailwind utilities generated by admin-console + host-shell builds do not cross the shadow boundary. Every consumer-side `class="max-w-sm hover:border-falcon-green-500 ..."` would be inert.
- The `wrapperClass` / `inputClass` / `labelClass` ergonomics that admin-console's wizard pages rely on (see `add-client-wizard/client-information-step` → Finance ID field) become impossible without exposing every internal node via `::part()` — that is a huge per-component API surface for a styling primitive Tailwind already provides for free.
- Theme Studio still works (CSS custom properties inherit), but per-instance utility overrides do not. The day-to-day developer flow gets worse.

### B. Light-only — rejected

Single variant, `shadow: false` for every component. Tailwind utilities flow in freely. Single TSX per component.

**Rejected because:**
- Encapsulation is gone. Any consumer's global CSS — including legacy rules in `falcon-web-platform-ui-old`, third-party host apps embedding Falcon as a remote, or aggressive admin overrides — can silently break component internals. The library loses its contract.
- React / Vue / plain-HTML consumers (the cross-framework future state per `project_falcon_ui_library`) get no encapsulation at all. The whole point of bringing Stencil into the stack is that it produces framework-agnostic components with encapsulation guarantees; throwing that away makes Stencil-vs-plain-Angular a wash.
- Component CSS would have to be loaded globally or shipped per-component via `<style>` injection, both of which create cascade-order and specificity wars with consumer CSS.

### C. CSS Shadow Parts (`::part()`) — partial, not a replacement

Stay Shadow-only and expose every styleable internal node as a `part="..."` attribute. Consumers then write `::part(input)` rules in their own CSS to override.

**Partial fix:**
- Solves deep customization, but does NOT solve utility-class consumption. A consumer cannot write `<falcon-input class="max-w-sm">` and have `max-w-sm` apply to the rendered shell — they have to write a global CSS rule targeting `::part(wrapper)` instead. This breaks the Tailwind-first DX admin-console relies on.
- Forces a per-component "parts API" to be designed, documented, and versioned — much higher API surface than the dual-render approach where the Tailwind variant simply exposes nodes the caller can class.
- Falcon does still expose `part="..."` selectively (see falcon-input's structural divs) for power users, but it is a complement to, not a replacement for, the Tailwind render path.

### D. Slots-only customization — partial, not a replacement

Keep Shadow-only and lean hard on `<slot>` insertion points so consumers project their own markup for prefix, suffix, icons, etc.

**Partial fix:**
- Slots cover composition (prefix icon, action button, suffix content), not customization (border color on hover, max-width, focus ring tweak). The Wave 1 / Wave 2 component set needs both.
- Too rigid: every visual tweak forces a slot, and slot proliferation is its own design-debt anti-pattern.

## Consequences

### Positive

- **Each consumer picks the variant that fits.** Tailwind-first apps default to `<falcon-X-tw>` (Light DOM) and get full utility-class flow. Encapsulation-first consumers pass `[useTailwind]="false"` and get `<falcon-X>` (Shadow DOM) with slot-based composition. Same wrapper API either way.
- **Tokens are the SSOT, parity is by construction.** Both variants resolve every visual value through `var(--falcon-X-*)`. Mutating one token in the Theme Studio updates both render paths simultaneously, in every framework, with no per-variant patch. This is the design that makes the Studio possible.
- **Future React/Vue wrappers get both variants automatically.** Stencil emits framework-agnostic custom elements. When the React (Wave 5) and Vue (Wave 6) wrappers ship, they implement the same tag-switcher pattern over the same two custom-element tags — no duplicate component logic, no second design system to maintain.
- **Per-instance token override works in both variants.** The `add-client-special-input` class on admin-console's Account Name field mutates `--falcon-input-*` tokens on a single host element; both the Shadow internal CSS and the Tailwind arbitrary-value utilities pick up the override through the same `:where()` selector chain.
- **Tailwind path supports `wrapperClass` / `inputClass` / `labelClass` ergonomics.** Caller-supplied classes land on real DOM nodes, supporting the admin-console hover-green-border pattern and similar per-instance utility tweaks without a single line of consumer SCSS.
- **Shadow path supports prop-driven feature toggles.** Booleans like `borderless`, `shadowless`, `flat`, `noFocusRing` reflect to host attributes (`:host([borderless])`) and toggle visual layers via CSS — clean, declarative, no class-string gymnastics.

### Negative

- **Two render paths to test.** Every component needs runtime probes against `<falcon-X>` AND `<falcon-X-tw>`. The Wave 2 Phase D "deep regression" sweep exists precisely to keep them honest.
- **Two TSX files per component to maintain.** Props, events, slots, ARIA wiring, lifecycle — kept in sync by hand. The component-creation strategy at `Brain Outputs/strategies/falcon-component-creation/04-FILE_TEMPLATES/` ships `stencil-shadow-component.tsx.template` and `stencil-tw-component.tsx.template` side-by-side so new components inherit the parity by default.
- **Shadow DOM body-portal quirks.** Components that float overlays (`falcon-dropdown`, `falcon-date-picker`, `falcon-multi-select`) need `appendTo='body'` plus a Light-DOM wrapper trick (the `<div class="h-0 overflow-visible">` parking spot pattern documented in [project_falcon_ui_core_layout_traps](../../../memory/project_falcon_ui_core_layout_traps.md)) to position correctly relative to siblings. The Tailwind variant sidesteps this; the Shadow variant pays the cost.
- **Tailwind-variant slot gaps.** The Tailwind path does not support every slot the Shadow path does. `<falcon-input-tw>` currently lacks `prefix` / `suffix` slots — consumers needing slots must use Shadow mode. This is a documented divergence in `falcon-input/API.md` and a known per-component upgrade target.
- **Stencil-only feature props.** `borderless`, `shadowless`, `flat`, `noFocusRing` exist only on the Shadow path because they are implemented via `:host([attr])` CSS rules. The Tailwind variant ignores them. Acceptable today because these are rare visual toggles; tracked as a follow-up if their usage grows.
- **Defense-in-depth Tailwind safelist.** Tailwind v4's static scanner cannot follow class strings split across `?:` ternaries inside library TS. Every app's `tailwind.css` needs `@source "../../../libs/falcon-ui-core/src/tailwind"`, `@source "../../../libs/falcon-ui-core/src/angular-wrapper"`, and explicit `@source inline("...")` for arbitrary-value utilities. This is build-config tax, not runtime cost, but it has to be remembered.

### Trade-offs accepted

- **2× TSX maintenance accepted in exchange for one source of design truth.** The cost is real; the alternative — two divergent component libraries (one Tailwind-y, one shadow-encapsulated) maintained independently — is worse. Tokens unify them, and the templates enforce the structural mirror so the per-component drift surface stays small.
- **Default to Tailwind, opt into Shadow.** Picking a sensible default (`useTailwind: true`) means 95% of usages don't think about the choice. The 5% who need slots or hard isolation flip one input. The cost of the choice lives at one boundary.
- **No third variant.** No "Tailwind but in Shadow" hybrid, no "Light DOM without Tailwind". Two paths, picked at one switch. Keeps the matrix from exploding.

## Verification

### Current state

- **Two variants exist for ~94 components.** `libs/falcon-ui-core/src/components/` contains 94 sibling folders covering the dual-render set: `falcon-input/` paired with `falcon-input-tw/`, `falcon-alert-dialog/` paired with `falcon-alert-dialog-tw/`, `falcon-dialog/` paired with `falcon-dialog-tw/`, `falcon-dropdown/` paired with `falcon-dropdown-tw/`, and so on across the production component set. (A few internal helpers like `falcon-toast-host` and a handful of Angular-only wrappers like `falcon-organization-hierarchy-tree-tw` ship a single variant by design.)

- **`<falcon-input>` is the OG dual-render component.**
  - Shadow source: `libs/falcon-ui-core/src/components/falcon-input/falcon-input.tsx` declares `@Component({ tag: 'falcon-input', styleUrl: 'falcon-input.css', shadow: true })` — every visual value reads `var(--falcon-input-*)`.
  - Tailwind source: `libs/falcon-ui-core/src/components/falcon-input-tw/falcon-input-tw.tsx` declares `@Component({ tag: 'falcon-input-tw', shadow: false })` and imports `falconInputWrapperClasses`, `falconInputControlClasses`, `falconInputLabelClasses`, etc. from `libs/falcon-ui-core/src/tailwind/input-tailwind-classes.ts` — the same Tailwind helper module used to be called from framework wrappers and is now collapsed into the Light-DOM component so framework wrappers stay thin.
  - The file header in `falcon-input-tw.tsx` documents the contract explicitly: "Tokens still flow through `--falcon-input-*` so a Studio that mutates tokens at runtime updates this component identically to `<falcon-input>`."

- **Angular wrapper as the switch point.** `libs/falcon-ui-core/angular-wrapper/components/falcon-input/falcon-input.component.ts` exposes `useTailwind: boolean` (default `true`) and renders either `<falcon-input>` or `<falcon-input-tw>` accordingly. Documented in `Brain Outputs/understanding/frontend/components/falcon-input/API.md`:
  > `useTailwind` `boolean` `true` **Render-path switch.** `true` → `<falcon-input-tw>` (Light DOM). `false` → `<falcon-input>` (Shadow DOM).

- **Tailwind variant is the production default.** Both proof points in admin-console (`apps/admin-console/src/app/features/organization-hierarchy/components/wizard-components/add-client-wizard/client-information-step/client-information-step.component.html`) exercise the dual path:
  - **Account Name** field — Shadow mode (`useTailwind=false`) — proves per-instance token override via `.add-client-special-input` class
  - **Finance ID** field — Tailwind mode (`useTailwind=true`, the default) — proves caller-supplied `wrapperClass="hover:border-falcon-green-500 ..."` utility-class flow into Light DOM

- **Template enforcement.** New components scaffold from `Brain Outputs/strategies/falcon-component-creation/04-FILE_TEMPLATES/stencil-shadow-component.tsx.template` and `stencil-tw-component.tsx.template` together. The component-creation doctrine (§5–§6 of `01-CANONICAL_PATTERN.md`) treats them as a pair, not a choice.

- **Token SSOT enforced per component.** Both variants read the same `falcon-X.tokens.css` contract. Per `feedback_shadow_is_token_ssot`: the Tailwind variant must reference the exact `--falcon-X-*` names the Shadow CSS reads, never invent tokens, never hardcode hex/px/ms. Parity audit drills (Wave 2 Phase A1) diff contract ↔ Shadow CSS ↔ Tailwind helpers side-by-side and flag any drift as a bug.

- **`<falcon-alert-dialog>` confirms the pattern at the composite tier.** The most recently shipped composite dialog (2026-05-13) follows the same shape — `falcon-alert-dialog.tsx` (Shadow) and `falcon-alert-dialog-tw.tsx` (Light) reading the same `falcon-alert-dialog.tokens.css`, picked by the Angular wrapper's `useTailwind` input. Production-ready, consumed by the `falcon-insufficient-balance-dialog` app-wrapper. Per its OVERVIEW: *"Both variants consume the same `falcon-alert-dialog.tokens.css` source of truth."*

- **Studio-readiness verified at the build level.** Mutating any `--color-falcon-*` or `--falcon-input-*` token at runtime updates Stencil-Shadow AND Tailwind-Light in every consuming framework simultaneously, because both resolve through the same `:where()` selector chain rooted in `libs/falcon/src/theme/falcon-tailwind-tokens.css`. See [project_falcon_ui_library](../../../memory/project_falcon_ui_library.md) ("Single source of truth" diagram).

### What would falsify this ADR

- A Theme Studio token mutation that updates one render path but not the other on a given component → token-contract violation; fix the variant that drifted, not the Studio.
- A consumer needing Tailwind utility flow that has to drop down to a hand-written Stencil component → either extend the Tailwind variant or document the gap and create an upgrade ticket; do not introduce a third render path.
- The cross-framework promise breaking when React or Vue wrappers ship — i.e. discovering the Stencil custom-element output doesn't carry through. Mitigated by the fact that this is Stencil's primary use case and the React/Vue wrappers will be tag-switchers, not duplicate component implementations.
