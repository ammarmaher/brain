*** ADR-007 — Why Falcon declares design tokens in Tailwind v4 `@theme` (CSS-native) rather than `@config` (legacy JS-style) ***
*** Status: accepted · Reversal severity: hard (218 tokens + 47 component-scoped files depend on the @theme contract) ***

# ADR-007 — Why Tailwind v4 `@theme` over `@config`

## Context

Tailwind v3 declared design tokens in a JavaScript config file — `tailwind.config.{ts,js}` exporting an object literal under `theme.extend.colors`, `theme.extend.spacing`, etc. Tooling, IDE intellisense, and the build pipeline all assumed JS was the source of truth.

Tailwind v4 (which Falcon adopted as part of the V0.2 theme migration — see [MEMORY] `feedback_v02_theme_adopted` and **ADR-002**) introduced two ways to declare tokens, and they are not equivalent:

| Mechanism | Where tokens live | Consumed by |
|---|---|---|
| **`@config`** (legacy bridge) | a `tailwind.config.{ts,js}` file referenced by `@config "…";` in the entry CSS | only Tailwind's utility generator — values are JS literals, not CSS variables |
| **`@theme`** (CSS-native) | a `@theme { … }` block inside the entry CSS, declaring `--<namespace>-*` custom properties | Tailwind's utility generator **and** every CSS consumer via `var(--…)` — the values literally are CSS variables |

The Falcon frontend has two parallel render paths for every Falcon component:

1. **Stencil Shadow DOM** — reads tokens via `var(--falcon-…)` because Shadow DOM cannot see utility classes from the outer document.
2. **Light DOM Tailwind variant** (the `-tw` companion components) — consumes tokens as utility classes (`bg-falcon-teal-700`, `rounded-md`, `shadow-falcon-popover`).

Both paths must paint with the **same** values from the **same** source. The choice of token declaration mechanism is therefore a structural decision, not a styling preference.

Falcon picked `@theme`. This ADR records that choice.

## Decision

**Design tokens are declared exclusively in CSS-native `@theme` blocks**, with the canonical block at `libs/falcon-theme/src/falcon-tailwind-tokens.css` (the SSOT) plus 47 per-component companion files at `libs/falcon-ui-tokens/src/components/*.tokens.css` that resolve their private `--falcon-<component>-*` namespaces against the SSOT.

**The `@config` directive is NOT used to declare tokens anywhere in the workspace.** The single surviving `@config` reference at `libs/falcon-theme/src/falcon-tailwind-tokens.css:8` is a documented v3→v4 **compatibility bridge** that re-enables `important: true` so utility classes outrank legacy `::ng-deep` overrides — it does not carry token values. See [BRAIN-OUT] `TOKEN_TAXONOMY.md` §2 for the verbatim documentation of this bridge.

Concretely: every color, spacing, radius, shadow, font-family, type size, easing/duration, breakpoint, and z-index value used by any Falcon UI surface must trace back to a `--<namespace>-*` line inside that single `@theme` block. Tailwind v4 auto-promotes those custom properties into utility classes (`--color-falcon-teal-700` → `bg-falcon-teal-700` / `text-falcon-teal-700` / `border-falcon-teal-700` / …), and Stencil templates reference the same custom properties directly via `var(--color-falcon-teal-700)`. One declaration, two consumption paths, zero drift.

## Alternatives considered

### A. `@config` with a TypeScript token file — **rejected**

Express tokens as a `tailwind.config.ts` exporting a typed token tree (e.g. `theme.extend.colors.falcon.teal['700'] = '#0d3f44'`), reference it from the entry CSS with `@config "./tailwind.config.ts"`. Rejected because:

- **Defeats the "CSS variables ARE the source of truth" principle.** Tokens authored as JS literals do NOT become CSS custom properties unless Tailwind explicitly re-emits them. Stencil's Shadow DOM components cannot read JS literals — they need `var(--…)` resolvable at runtime. Going via `@config` forces a separate `--falcon-…` declaration block alongside the JS file, which immediately produces two sources of truth.
- **Breaks the Stencil ↔ Tailwind variant parity.** Each Falcon component ships a Shadow-DOM path and a Tailwind Light-DOM path. Both paths must paint identically. With `@config`, the Tailwind path renders against compiled JS values while the Shadow path renders against whatever `--falcon-…` block was authored to mirror it — and the two will drift the moment a single value changes in one place but not the other.
- **No diff-friendly review.** A `.ts` token file looks like JavaScript in code review; a `@theme` block looks like a values-only design spec. The 218-token spec is auditable at-a-glance in `@theme` form (see [BRAIN-OUT] `TOKEN_TAXONOMY.md`'s tables, all generated by reading the `@theme` block directly).
- **JS runtime overhead for tokens.** Pure-CSS toolchains (Stencil's CSS pipeline, the per-component `*.tokens.css` companions, Studio's runtime theme overrides) would need to import a JS bundle just to read color values. With `@theme`, the same `.css` file is the source for all of them.

### B. Both `@config` and `@theme` together — **rejected**

Keep a `tailwind.config.ts` for "JS-side knobs" (plugins, important, content globs) **and** declare token values in a `@theme` block. Allow some tokens to live in JS for "complex" cases. Rejected because:

- **Split-brain token state.** Two declaration sites means two answers to "where is `--color-falcon-teal-700` defined?", with no enforcement that they agree.
- **Permanently fragile.** Whichever declaration "wins" depends on Tailwind v4's internal resolution order, which is an implementation detail (the v3→v4 migration guide explicitly calls this an evolving surface). A correct build today becomes a wrong build after a Tailwind point release.
- **Tooling defeated.** The pre-finish grep gate (see [MEMORY] `feedback_no_inline_styles_tokens_only` and **R-FE-004**) verifies token discipline by reading CSS — it cannot reason about JS token trees without a separate parser. Keeping `@theme` as the sole declaration site keeps the gate cheap and total.

A bounded exception is already documented and tolerated: the **`@config` compatibility bridge** at line 8 of the SSOT exists solely to forward `important: true` from a near-empty `tailwind.config.js`. This bridge declares zero tokens. If Tailwind v4 grows a native `@important` directive, this bridge is removed in one line.

### C. Custom PostCSS plugin feeding tokens from a custom format — **rejected**

Author a Falcon-specific token format (e.g. YAML, JSON, or a custom DSL) and write a PostCSS plugin to compile it into both Tailwind theme values and CSS custom properties. Rejected because:

- **Too much custom infrastructure for zero net benefit.** Tailwind v4's `@theme` block already does this, natively, at the same point in the pipeline, with zero maintenance burden on Falcon.
- **Tooling tax.** Every contributor would need to learn the Falcon-specific format. Every IDE plugin, Stencil sourcemap, Tailwind language server, and CSS linter would need a custom integration. None of these tools knows about a bespoke DSL out of the box.
- **Lock-in worse than `@config`'s.** A custom plugin is owned by Falcon forever; `@config` is at least a Tailwind-standard escape hatch.
- **No portability win.** A custom DSL doesn't make tokens any more portable than the `@theme` block already is — CSS custom properties are already a W3C standard consumable by any toolchain.

### D. Per-app `tailwind.config.ts` — **rejected**

Declare tokens once per app (`apps/host-shell/tailwind.config.ts`, `apps/admin-console/tailwind.config.ts`, `apps/management-console/tailwind.config.ts`) with shared inheritance via a `preset`. Rejected because:

- **3× duplication, 3× drift risk.** Three Nx apps share one design system. The day someone tweaks `--color-falcon-teal-700` in one app and forgets the other two is the day Falcon's identity bifurcates per app. (We already lived through this with the pre-V0.2 short-alias families — see [MEMORY] `project_token_unification_plan`'s ~700-edit scope.)
- **Defeats the library/app split.** Tokens are a library concern (the design system), not an app concern (page routing, layout, feature wiring). Putting tokens in apps mis-locates the abstraction.
- **Module Federation hostility.** `host-shell` loads `admin-console` and `management-console` as remote MF entries. Each remote ships its own bundle; per-app token declarations mean per-bundle token state, which is unsafe across the MF boundary (a remote rendering against its own `--color-falcon-teal-700: #DEAD00` would visibly disagree with the host). With a single `libs/falcon-theme` import, every app and every remote resolves the same custom property tree at runtime against the host document's stylesheet.
- **Pre-finish gate complexity.** The grep gate would have to walk three trees and reconcile them; with one SSOT it walks one file.

## Consequences

### Positive

- **CSS-native — works in pure-CSS toolchains.** Stencil's `*.tokens.css` companion files, Studio's runtime theme overrides, dark-mode `:where(.app-dark, …)` overrides at lines 392–458 of the SSOT, and every component-scoped namespace at `libs/falcon-ui-tokens/src/components/` all merge cleanly with the same `var(--…)` references. No JS runtime is needed to read tokens.
- **Human-scannable and diff-friendly.** The `@theme` block at `falcon-tailwind-tokens.css:15-367` reads as a values-only design spec — 218 lines of `--name: value;` pairs grouped by family. PR review for a token change shows exactly one line in the diff with no JS-config noise.
- **Per-component token files merge cleanly with the canonical theme.** All 47 files at `libs/falcon-ui-tokens/src/components/*.tokens.css` resolve their private `--falcon-<component>-*` namespaces against the SSOT — e.g. `button.tokens.css` defines `--falcon-button-bg-primary: var(--color-falcon-teal-700)`. The component layer is a **theming indirection** layered over the SSOT, not an independent token set. Change the SSOT teal and every component repaints. (See [BRAIN-OUT] `TOKEN_TAXONOMY.md` §11.)
- **Stencil ↔ Tailwind variant parity is free.** Shadow DOM components and Light DOM Tailwind variants both bottom out at the same `var(--…)` resolved from the same document-level stylesheet. The dual render-path doctrine ([MEMORY] `project_falcon_ui_library`) only works because of this.
- **No JS runtime required for tokens.** Static analyzers, the pre-finish grep gate, Stencil's CSS bundler, and Studio's runtime swap all read the same `.css` file. None of them needs to spin up Node to evaluate a config module.
- **Module Federation safe.** Host and remotes load `@falcon/theme` once at the host document level; every MF remote inherits the same custom-property tree. No per-remote token bundle, no per-remote drift.

### Negative — Tailwind v4 quirks we accept

- **The `--<prefix>-*` naming map is opinionated, and the prefix MUST match Tailwind's known map for auto-promotion to fire.** Tailwind v4 only auto-promotes tokens whose CSS-variable name matches its known utility prefix set: `--color-*` → color utilities, `--spacing-*` → spacing utilities, `--radius-*` → `rounded-*`, `--shadow-*` → `shadow-*`, `--text-*` → `text-*`, `--font-*` → `font-*`, `--ease-*` → `ease-*`, `--duration-*` → `duration-*`, `--z-*` → `z-*`, `--breakpoint-*` → responsive prefixes, `--animate-*` → `animate-*`, `--tracking-*` → `tracking-*`, `--background-image-*` → `bg-*` (image), and the line-height prefix `--leading-*` → `leading-*`. Anything with a custom `--falcon-*` prefix (e.g. `--falcon-size-*`, `--falcon-border-width-*`, `--falcon-stepper-*`, `--falcon-icon-*`, `--transition-falcon-row`) is declared inside `@theme` but functions as a plain CSS variable — components consume it via `var()`, not via utility classes. This is intentional for component-private values, but it surfaces as **Gap G-06** (see Open questions) for the leading scale.
- **`--falcon-leading-*` is the canonical example of the quirk.** Five line-height tokens at lines 150–154 of the SSOT (`--falcon-leading-{tight,snug,normal,relaxed,loose}`) are declared inside `@theme` but **do NOT auto-promote** to `leading-*` utilities because Tailwind v4 reads `--leading-*` (not `--falcon-leading-*`). Components consume them only via `var(--falcon-leading-*)` in Stencil and per-component CSS. This is documented in [BRAIN-OUT] `TOKEN_TAXONOMY.md` §4.3 and §12 (the ⛔ NOT auto-promoted row), and surfaced as Gap G-06 in §14.
- **`@theme` requires the entry CSS to load before any consumer can paint with utility classes.** This is also true of `@config`, but new contributors sometimes assume tokens are "ambient" — they are not. Apps must `@import "@falcon/theme"` (or transitively load it via `libs/falcon-theme/src/index.css`) before any Falcon component renders.

### Trade-offs accepted

- We accept that adding a **new** namespace family that Tailwind v4 doesn't already know about means the new tokens won't get auto-promoted utilities — they'll be CSS-variable-only. For Falcon, that has been fine for `--falcon-size-*`, `--falcon-border-width-*`, `--falcon-stepper-*`, and component-private namespaces. When auto-promotion **is** wanted, the token must use the standard prefix (this is the open question for `--falcon-leading-*`).
- We accept that one developer convenience of `@config` — using JS spreads / loops to programmatically generate token trees — is unavailable in `@theme`. The 218-token block is hand-authored. We consider this a feature: every token is a named, reviewed design decision rather than a computed artifact.

## Open questions

- **Gap G-06 — should `--falcon-leading-*` be renamed to `--leading-*`?** Renaming the five line-height tokens (`tight 1.2`, `snug 1.3`, `normal 1.4`, `relaxed 1.5`, `loose 2.1`) from `--falcon-leading-*` to `--leading-*` would unlock Tailwind v4 auto-promotion of `leading-{tight,snug,normal,relaxed,loose}` utility classes — currently those utilities don't exist and templates must use inline `style="line-height: var(--falcon-leading-snug)"` or compose `--falcon-leading-*` from per-component CSS. The trade-off is namespace consistency: every other Falcon-specific token uses the `--falcon-` prefix; dropping it for `leading` introduces an exception. Recommended follow-up: a small ADR-row deciding "rename (auto-promote) vs document the var-only consumption pattern explicitly". Tracked at [BRAIN-OUT] `TOKEN_TAXONOMY.md` §14 G-06.
- **Should other `--falcon-*`-prefixed tokens that **could** map to a Tailwind utility family (e.g. `--falcon-border-width-*` → `border-*`) also be renamed?** Same trade-off, smaller blast radius (4 tokens). Worth scoping alongside G-06.
