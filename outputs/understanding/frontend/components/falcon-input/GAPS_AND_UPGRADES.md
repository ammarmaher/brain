# falcon-input — GAPS AND UPGRADES

## Known incomplete capabilities

1. **No multi-line variant** — `falcon-input` is single-line only. Use `falcon-angular-textarea` for multi-line.
2. **`prefix` / `suffix` slots are Shadow DOM only.** The Light DOM `<falcon-input-tw>` does not yet expose addon slots. Consumers needing icon prefixes in Light DOM mode have to compose externally with a wrapper div, which breaks the shared-border look.
3. **`autoFocusOnMount` is Stencil-only.** The Angular wrapper does not forward this prop — it has its own `nativeInputRef?.focus()` path but no autofocus on init.
4. **No built-in validation pipe** — error display relies on the consumer passing `errorMessage`. There is no async-validator integration, no debounced validation, no built-in regex / email / URL check inside the component.
5. **No `min` / `max` for `type='number'`.** Use `falcon-angular-input-number` instead — that component has `min` / `max` / `step` / `decimals` props.
6. **No `pattern` prop** — `minlength` / `maxlength` exist, but native `pattern` is not exposed.
7. **No password-strength indicator** — that is `falcon-angular-password`'s job.
8. **No `inputmode` prop** — affects mobile keyboard hint. Only `type` is exposed.
9. **No way to slot a button into the field on the Light DOM path** — `<falcon-email-field>` and `<falcon-phone-field>` solve this by being dedicated composite components rather than slotting into `<falcon-input>`.

## a11y status

- `aria-invalid` is set from `isFieldInError` (utility in `falcon-input.utils.ts`).
- `aria-describedby` joins helper / error ids via `joinIds()`.
- `aria-required` is set from `required`.
- `aria-label` on the clear button comes from `clearAriaLabel`.
- ARIA live-region announcement of error changes is **NOT** yet implemented. Screen readers may not pick up newly-set `errorMessage` content.
- Focus management on `clear()` re-focuses the input (Light DOM handler in `handleNativeClear`).

## Test coverage

- `falcon-input.spec.ts` and `falcon-input.e2e.ts` exist next to the source — coverage is component-level. No cross-framework integration spec at the workspace root.

## Performance notes

- `methods()` not `computed()` is deliberate — see `falcon-input.component.ts:106-108`. Worth a note in any future migration to signals-only API.
- `OnPush` + signal-based internal `value` keeps re-renders cheap. The wrapper does not import RxJS at runtime.

## Upgrade ideas (out of scope for this audit; capture only)

- Forward `autoFocusOnMount` from the Angular wrapper to the rendered tag.
- Add `prefix` / `suffix` slots to `<falcon-input-tw>` to close the parity gap with Shadow DOM.
- Add an optional `inputmode` prop.
- Optional `pattern` prop with `aria-invalid` wiring.
- Live-region announcement of `errorMessage` changes (`aria-live="polite"` on the error span).
- Async-validator-friendly debounced commit mode (`commitMode: 'change' | 'blur' | 'debounced(ms)'`).
- Optional regex / email / URL built-in validation that emits a `falcon-validation` event (still leaving the consumer in control of the `errorMessage` content).
- Consider migrating the wrapper's internal `value` / `disabled` signals to `model()` / `input()` signal-based APIs once Angular 21 patterns stabilise.

## Cross-framework drift checks

- React wrapper at `libs/falcon-ui-react/src/` is auto-generated by Stencil's `reactOutputTarget` — should track 1:1 with the Stencil core.
- Vue wrapper at `libs/falcon-ui-vue/src/` is generated by `libs/falcon-ui-core/generate-vue-proxies.cjs` — verify after any prop addition that the script regenerated the Vue proxy.
- The Tailwind helpers (`falconInputWrapperClasses` etc.) are imported by Angular AND by the Light DOM Stencil tag — same source ensures identical class output across frameworks.
