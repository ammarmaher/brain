# falcon-dropdown — GAPS AND UPGRADES

## Known incomplete capabilities

1. **No per-item template slot** — Wave 4 follow-up flagged at multiple call sites (e.g. `user-role-status-step.component.ts`). Consumers must translate option labels upfront and re-translate on language change. Live language switch isn't supported on rendered labels until this slot lands.
2. **No multi-select.** This is by design — use `falcon-angular-multi-select` for that case. Worth surfacing in the registry so consumers don't go looking.
3. **No async / lazy option loading.** The component takes a static `options: FalconDropdownOption[]` array. There is no built-in "loading more" indicator, no infinite-scroll, no server-side search debouncing.
4. **`falcon-search` event is not debounced** — the component emits on every keystroke. Consumers must debounce themselves before hitting the server.
5. **No virtualization.** Large option lists (1000+) will render every DOM node. The panel `max-height` token controls visible viewport, but DOM size grows linearly.
6. **No group-header customisation.** When `option.group` clusters options, the header text and styling are internal — no slot per group.
7. **No keyboard tab-through into the panel.** Tab moves focus past the trigger to the next form field once the panel is closed. Inside the panel, navigation is Arrow / Enter / Esc only.
8. **`emptyMessage` is a string, not a template.** Consumers can't slot a custom "no results" rich layout.
9. **The slotted `options` slot is documented in the component (`<slot name="options">`) but the default rendering is non-overridable per-option — it's all-or-nothing.** Real custom rendering would need a per-option template plus a render-function prop.

## a11y status

- `aria-haspopup="listbox"` on the trigger.
- `aria-expanded` toggles with `open`.
- `aria-activedescendant` on the listbox points to the active option id.
- `aria-required` mirrors `required`.
- Search input has `aria-label="Search options"` (default) — TODO confirm i18n.
- `aria-invalid` mirrors `isFieldInError(state, errorMessage)`.
- Listbox panel uses `role="listbox"` + per-option `role="option"` + `aria-selected`.

## Test coverage

- The component does not ship a `.spec.ts` / `.e2e.ts` file alongside (unlike `falcon-input`). Coverage gap.

## Performance notes

- Type-ahead buffer drains after 600ms — change tracker uses `setTimeout`, cleared on `disconnectedCallback`.
- `filterOptions` (in `falcon-dropdown.utils.ts`) does a substring match. Case-insensitive but no fuzzy / accent-folding.
- `moveActiveIndex` skips disabled options correctly.

## Upgrade ideas (out of scope here; capture only)

- Add a per-option `ng-template` slot so consumers can render rich options (avatar + label + meta line + badges) without a fork.
- Add `loading: boolean` + `loadingMessage` + a built-in skeleton row in the panel.
- Add `searchDebounce: number` and an internally-debounced `falcon-search` event.
- Add `virtualize: boolean` + `itemHeight: number` props using a fixed-size virtual list.
- Make `emptyMessage` accept a `TemplateRef` (Angular wrapper only) or a slot (Stencil).
- Add `groupBy` + per-group template slot.
- Track Wave 4 backlog item for the per-item slot — currently the biggest UX gap.
- `.spec.ts` + `.e2e.ts` to bring it to parity with `falcon-input`.

## Cross-framework drift checks

- React wrapper at `libs/falcon-ui-react/src/` is auto-generated by Stencil's `reactOutputTarget` — should track 1:1.
- Vue wrapper at `libs/falcon-ui-vue/src/` is generated via the Vue proxy script — verify after prop changes.
- Tailwind helpers (`falconDropdownTriggerClasses` etc.) are shared between Angular wrapper, Light DOM Stencil tag, React + Vue — single source.
